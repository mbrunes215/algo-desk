# Algorithmic Trading Desk - Project Structure

## Overview
Complete production-quality Python trading desk project with full type hints, comprehensive monitoring, risk management, and testing infrastructure.

## Core Modules

### 1. Monitoring Module (`monitoring/`)

#### dashboard.py
- **TradingDashboard**: Terminal-based dashboard with real-time updates
  - Methods: `refresh()`, `display()`, `get_status_summary()`
  - Shows: positions, P&L, strategy status, recent trades, system health
  - `generate_html_dashboard()`: Creates self-contained HTML dashboard file
  - Features: Professional styling, responsive layout, live updates

#### alerts.py
- **AlertManager**: Comprehensive alert system with email support
  - Alert levels: INFO, WARNING, CRITICAL
  - Rate limiting: Configurable max alerts per hour
  - Email integration: SMTP configuration, TLS support
  - Methods:
    - `send_alert()`: Send alert at specified level
    - `send_email_alert()`: Send email notifications
    - `check_and_alert()`: Conditional alert sending
    - Specialized methods: `alert_kill_switch_triggered()`, `alert_connection_lost()`, `alert_strategy_error()`, `alert_pnl_threshold()`
  - Alert history tracking and summary statistics

#### health_check.py
- **HealthChecker**: Monitors system and service health
  - Checks:
    - IBKR connection (TCP socket test)
    - Kalshi API connectivity (HTTP health endpoint)
    - Database accessibility (SQLite file access)
    - Strategy heartbeats (timeout-based)
    - Disk space (configurable minimum)
    - Memory usage (configurable threshold)
  - Returns: `HealthReport` dataclass with status for each check
  - Methods: `run_all_checks()`, `check_ibkr()`, `check_kalshi()`, `check_database()`, `check_disk_space()`, `check_memory()`, `check_strategy_heartbeats()`

### 2. Data Module (`data/`)

#### pipelines/market_data.py
- **MarketDataPipeline**: Fetches and stores market data
  - Sources: IBKR (stocks), Kalshi (prediction markets)
  - Features:
    - Incremental updates (only fetch new data)
    - SQLite storage with automatic schema creation
    - Historical data retrieval
    - Metadata tracking for last update times
  - Methods:
    - `fetch_ibkr_historical()`: Fetch stock data with duration/bar_size
    - `fetch_kalshi_markets()`: Fetch prediction market data
    - `store_data()`: Insert/update data in database
    - `get_historical()`: Query historical data by date range
  - Includes mock data generation for testing

#### storage/models.py
- **SQLAlchemy ORM Models**:
  - `Trade`: Single trade execution record
    - Fields: strategy, symbol, type (BUY/SELL), quantity, price, status, order_id
    - Status: PENDING, SUBMITTED, FILLED, PARTIAL_FILL, CANCELLED, REJECTED
  - `Position`: Current or historical position
    - Fields: strategy, symbol, quantity, average_price, current_price, status
    - Status: OPEN, CLOSED, LIQUIDATED
  - `MarketData`: Historical market data
    - Fields: source (IBKR/Kalshi), symbol, timestamp, OHLCV data
  - `StrategySignal`: Trading signals generated by strategies
    - Fields: strategy, symbol, signal_type (BUY/SELL/HOLD), confidence, signal_data
  - `Alert`: Alert events
    - Fields: level, message, context_data, timestamps
  - `SessionFactory`: Manages database connections and session creation

### 3. Main Application (`main.py`)

- **TradingDeskApplication**: Core application class
  - Features:
    - YAML config file loading
    - Environment variable support (.env)
    - Graceful signal handling (SIGINT/SIGTERM)
    - Component initialization (database, alerts, health checker, dashboard)
    - Strategy initialization and filtering
    - Main event loop with periodic health checks
    - Dashboard refresh and display
  - CLI Arguments:
    - `--config`: Path to YAML config (default: config.yaml)
    - `--paper`: Enable paper trading mode (no real orders)
    - `--strategy`: Run only specific strategy by name
    - `--dashboard`: Display live dashboard
  - Methods:
    - `setup_logging()`: Configure file and console logging
    - `load_config()`: Load YAML and environment config
    - `initialize_components()`: Initialize all subsystems
    - `initialize_strategies()`: Load trading strategies
    - `run_health_checks()`: Execute health monitoring
    - `refresh_dashboard()`: Update and display dashboard
    - `main_loop()`: Main trading event loop
    - `shutdown()`: Graceful shutdown

### 4. Test Suite (`tests/`)

#### conftest.py
- **Pytest Fixtures** (shared across all tests):
  - `test_database`: Creates/cleans test database
  - `db_session`: Database session for ORM testing
  - `sample_trade()`: Sample trade data
  - `sample_position()`: Sample position data
  - `sample_market_data_point()`: Single market data point
  - `sample_market_data_points()`: Series of market data points
  - `alert_manager()`: AlertManager instance
  - `health_checker()`: HealthChecker instance
  - `mock_positions()`, `mock_trades()`, `mock_strategy_status()`: Dashboard mock data
  - `mock_system_health()`: Health check mock data

#### test_risk.py
- **Risk Management Tests** (with mock classes):
  - `MockKillSwitch`: Kill switch implementation
    - Tests: trigger, is_triggered, reset
  - `MockDailyLimitManager`: Daily loss/profit limit enforcement
    - Tests: loss limit exceeded, profit target reached, daily reset
  - `MockPosition`: Position P&L calculations
    - Tests: unrealized P&L, unrealized P&L percent
  - `MockPositionManager`: Multi-position management
    - Tests: add, close, total P&L, price updates, max size enforcement

#### test_strategies.py
- **Strategy Tests**:
  - `WeatherStrategy`: Weather probability calculations
    - Methods: `calculate_weather_probability()`, `generate_signal()`
    - Tests: probability calculations, signal generation (BUY/SELL/HOLD)
  - `EconomicStrategy`: Economic surprise calculations
    - Methods: `calculate_surprise_distribution()`, `generate_signal()`
    - Tests: surprise calculations, signal generation, sensitivity
  - `MockMarketDataPoint`: Market data utilities
    - Properties: `daily_return`, `intraday_range`
    - Tests: Multiple signal generation with market data

## Usage Examples

### Run with Live Trading
```bash
python main.py
```

### Run in Paper Trading Mode
```bash
python main.py --paper
```

### Run Specific Strategy Only
```bash
python main.py --strategy WeatherStrategy
```

### Display Dashboard
```bash
python main.py --dashboard
```

### Run Tests
```bash
pytest tests/
pytest tests/test_risk.py -v
pytest tests/test_strategies.py::TestWeatherStrategy -v
pytest tests/ -m unit
```

## Configuration

Create `config.yaml`:
```yaml
database:
  url: sqlite:///trading.db

smtp:
  host: smtp.gmail.com
  port: 587
  user: your-email@gmail.com
  password: your-password

alerts:
  to_emails:
    - alerts@example.com
  max_per_hour: 10

ibkr:
  host: 127.0.0.1
  port: 7497

kalshi:
  api_url: https://api.kalshi.com
  api_key: your-api-key

strategies:
  WeatherStrategy:
    enabled: true
    confidence_threshold: 0.7
  EconomicStrategy:
    enabled: true
    sensitivity: 1.0
```

Environment variables in `.env`:
```
DATABASE_URL=sqlite:///trading.db
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=user@gmail.com
SMTP_PASSWORD=password
ALERT_EMAILS=alerts@example.com
IBKR_HOST=127.0.0.1
IBKR_PORT=7497
KALSHI_API_URL=https://api.kalshi.com
KALSHI_API_KEY=your-key
```

## Logging

- **Console Output**: INFO level and above
- **File Output**: DEBUG level and above
- **Location**: `logs/trading_desk.log`
- **Rotation**: 10 MB max file size, 5 backup files
- **Format**: Timestamp, logger name, level, function:line, message

## Architecture Notes

- Modular design with clear separation of concerns
- Each module is independently testable
- Type hints enable IDE autocomplete and static analysis
- Fixture-based testing reduces duplication
- Mock classes eliminate external dependencies in tests
- SQLite for simplicity (easily swappable with PostgreSQL)
- SMTP-based alerts (configurable)
- Clean architecture patterns for extensibility
